# Wallet strategy for a Pharos automation agent

## The short answer
For an **automated agent** you do NOT use a consumer wallet app (MetaMask/Rabby/OKX).
Those need human clicks and can't be driven headlessly. The correct wallet for agent
automation on any EVM L1 (Pharos included) is a **dedicated EOA keypair** generated by
`cast wallet` (or viem's `privateKeyToAccount`), with the private key in an env var / file.
This is exactly what Foundry and the x402 client both use — no browser, no extension.

We already generated two throwaway testnet keypairs (gitignored, chmod 600):

| Role | Env file | Address | Needs |
|------|----------|---------|-------|
| Seller / service agent (deploys SafeRegistry, receives USDC) | `.env.testnet` (`PRIVATE_KEY`/`DEPLOYER`) | `0x3F5a89E3CA45885880dAa0d4f1dA646170abD3bb` | PHRS gas |
| Caller / buyer agent (pays USDC to invoke) | `.env.caller` (`CALLER_PRIVATE_KEY`/`CALLER_ADDRESS`) | `0x4C1a8fE133E07Ab70aD2E882C048C4eCB056b1aE` | testnet USDC (+ maybe gas) |

## Why two wallets
Agent-to-agent means one agent pays another. The buyer signs USDC payment authorizations;
the seller receives. Keeping them separate makes the A2A loop real and the on-chain flow
legible to judges.

## Fund these (the only manual step)
Send testnet PHRS to BOTH addresses from a faucet (see TESTPLAN.md). For the x402 loop the
caller also needs testnet **USDC** (`0xE0BE08c77f415F577A1B3A9aD7a1Df1479564ec8`) — obtain
from the Pharos testnet faucet/portal or by swapping PHRS→USDC on a testnet DEX.

## For holding real winnings (PROS) — different wallet
Hot automation keys should never hold anything valuable. For real $PROS rewards, use a
**hardware wallet (Ledger)** or a fresh MetaMask/Rabby account with Pharos **mainnet** added
(chainId 1672, RPC https://rpc.pharos.xyz), kept entirely separate from these testnet keys.

## Security
- These keys are **testnet-only**. Never send mainnet assets to them.
- Keys live only in `.env.testnet` / `.env.caller` (gitignored, chmod 600) and are passed
  explicitly to `cast`/viem — never logged, never committed.
